## Why This Structure?

| Directory | Rationale |
|---|---|
| `src/UI/` | Clear boundary for UI-facing code (Presentation tier). Only this tier knows about HTTP, JWT, Angular. |
| `src/API/` | Internal API host. References Application + Infrastructure for DI wiring only. |
| `src/Core/Application/Features/` | **Feature-based** organization. Each feature is self-contained with Commands + Queries + Validators + Services. Easy to find and modify. |
| `src/Core/Domain/` | **Zero dependencies**. Pure business rules, entities, interfaces. Can be unit tested without any infrastructure. |
| `src/Infrastructure/` | All "details" � databases, external services, file I/O, encryption. Implements Domain interfaces. |
| `src/Contracts/` | Shared DTOs and settings. Referenced by both Presentation and Core tiers. |
| `tests/` | Mirror of `src/` structure. Each test project targets one source project. |
| `Directory.Build.props` | Central configuration for all projects (nullable, implicit usings, TFM). |
| `Directory.Packages.props` | **Central Package Management** � all NuGet versions managed in one file. |

## Directory.Build.props

```xml
<Project>
  <PropertyGroup>
    <TargetFramework>net8.0</TargetFramework>
    <Nullable>enable</Nullable>
    <ImplicitUsings>enable</ImplicitUsings>
    <TreatWarningsAsErrors>true</TreatWarningsAsErrors>
    <AnalysisLevel>latest-recommended</AnalysisLevel>
  </PropertyGroup>
</Project>
```
